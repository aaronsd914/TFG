[BD T1] — Mentionar "Sistema de Información" en Cap. 1:
  En la introducción del Cap. 1 puedes añadir una oración que califique
  FurniGest como "Sistema de Información para la Gestión" (SIG), ya que
  el Tema 1 de BD define exactamente este concepto: captura, almacena,
  procesa y distribuye información para apoyar la toma de decisiones.
  Coste: 2 líneas.

[BD T7b / Normalización] — Mencionar normalización en el modelo de datos:
  Indicar que el esquema de la BD cumple la 3ª Forma Normal (3FN)
  (o Forma Normal de Boyce-Codd si aplica). Esto demuestra que el
  diseño evita redundancias y anomalías de actualización/inserción/borrado.
  Coste: 1 párrafo breve.

[SD T1 — Teorema CAP] — Añadir referencia al teorema CAP:
  Breve mención en la sección de arquitectura: dado que FurniGest
  usa PostgreSQL (CP: Consistency + Partition tolerance sacrificando
  Available en particiones de red), se puede mencionar este compromiso.
  Coste: 2-3 líneas.

[IPO T6 — Evaluación de IUs] — Añadir en Cap. 4 (Resultados):
  Mencionar que la UI fue evaluada mediante el método de
  "evaluación heurística" (Tema 6 IPO) usando las heurísticas de

  de errores, etc. Si se hizo alguna sesión de prueba con usuarios reales,
  mejor aún — ese es el método empírico de evaluación.
  Coste: 1 párrafo en Cap. 4.

[IPO T5 — Accesibilidad] — Citar accesibilidad en RNF:

Ordenadas de mayor a menor impacto académico y alineadas con temarios:

────────────────────────────────────────────────────────────────────────────
1. AUTENTICACIÓN JWT + ROLES DE USUARIO
   Asignatura: SD T2 (Spring Security / protección de endpoints)
               IS T2 (RNF de seguridad), AIS T1 (pruebas de seguridad)
   Qué es: FastAPI Security con OAuth2PasswordBearer + JWT (PyJWT).
   Por qué: El mayor gap de seguridad del proyecto. Sin auth, cualquiera
            con la URL puede leer o modificar todos los datos.
   Roles sugeridos: admin, vendedor, repartidor.
   Coste estimado: 2-3 días de trabajo + 6-8 tests nuevos.
   En la memoria: nuevo subsección "Seguridad y control de acceso",
                  nuevo RNF-07, tests de autenticación.

────────────────────────────────────────────────────────────────────────────
2. MIGRACIONES DE BD CON ALEMBIC
   Asignatura: BD T6/T7 (diseño relacional, evolución del esquema)
               AIS T2 (mantenimiento y evolución del software)
   Qué es: Herramienta de migraciones para SQLAlchemy. Permite
           modificar tablas ya existentes sin perder datos.
   Por qué: create_all() solo crea tablas que no existen; no puede
            añadir columnas a tablas ya existentes en producción.
           Es una técnica fundamental de mantenimiento de BD (BD T6).
   Coste estimado: medio día para configurar + 1 migración inicial.
   En la memoria: subsección en CI/CD sobre "migraciones automáticas".

────────────────────────────────────────────────────────────────────────────
3. TESTS E2E CON PLAYWRIGHT
   Asignatura: AIS T1.3 (Pruebas de sistema/E2E), Práctica 2 AIS
   Qué es: Automatizar el navegador para verificar flujos completos
           de usuario (crear venta → cambiar estado → ver en contabilidad).
   Por qué: Completa la pirámide de pruebas con el nivel más alto.
           AIS T1 trata exactamente este tipo de prueba (Selenium/Playwright).
   Flujos críticos sugeridos:
     - Crear un albarán y verificar que aparece en la lista
     - Avanzar estado FIANZA → ENTREGADO y verificar el movimiento
   Coste estimado: 1 día + integración en el pipeline de CI.
   En la memoria: nueva fila en tabla de pruebas, diagrama de pirámide.

────────────────────────────────────────────────────────────────────────────
4. COBERTURA DE CÓDIGO (pytest-cov + badge en README)
   Asignatura: AIS T1.4 (Calidad del software), AIS T1.5 (SCA/SonarQube)
   Qué es: pytest --cov genera informe línea a línea de cobertura.
           Se puede subir a Codecov o mostrar en GitHub Actions.
   Por qué: Métrica objetiva de calidad interna (AIS T1.4).
            SonarQube (AIS T1.5) usa la cobertura como KPI clave.
   Coste estimado: 2 horas (pip install pytest-cov + config en CI).
   En la memoria: añadir % de cobertura en tabla métricas Cap. 4.

────────────────────────────────────────────────────────────────────────────
5. GESTIÓN DE STOCK / INVENTARIO
   Asignatura: BD T4/T5 (modelo E/R, modelo relacional)
               IS T2 (requisitos funcionales no implementados)
   Qué es: El campo `stock` ya existe en `productos` pero no se descuenta
           al crear líneas de albarán ni se controla el stock mínimo.
   Sugerencias:
     - Descontar stock al crear cada línea de albarán
     - Reposición de stock al crear un movimiento de compra a proveedor
     - Alertas visuales cuando stock < umbral mínimo configurable
   Coste estimado: 1 día (lógica en api/albaranes.py + 4-5 tests).
   En la memoria: nuevo RF-14, nueva entidad o atributo en tabla E/R.

────────────────────────────────────────────────────────────────────────────

   Por qué: La observabilidad es una propiedad de los sistemas distribuidos

7. RATE LIMITING EN ENDPOINTS DE IA/STRIPE

9. MÓDULO DE PRESUPUESTOS (estado previo al albarán)

11. APLICACIÓN MÓVIL / PWA

    Coste estimado: 2-3 horas para PWA basic (Vite PWA plugin).

12. BASE DE DATOS NoSQL / CACHÉ REDIS para analíticas
    Asignatura: SD T5 (Sistemas de persistencia distribuidos),
                SD T2.4 (Spring Data menciona Redis y MongoDB)
    Qué es: Cachear los resultados de /api/analytics/summary en Redis
            (TTL de 60 segundos). Las analíticas son costosas en BD.
    Por qué: SD T5 trata persistencia distribuida y caché. El uso de
            Redis demuestra conocimiento de bases de datos en memoria
            y sistemas de persistencia distribuidos.
    Coste estimado: 1 día.
    En la memoria: nueva subsección "Optimización de rendimiento".

13. ANALÍTICAS PREDICTIVAS (ARIMA / serie temporal)
    Asignatura: BD T1 (SI para toma de decisiones), IPO T4 (utilidad)
    Qué es: Usar los datos históricos de movimientos para predecir
            ventas del próximo mes (Prophet de Meta o statsmodels ARIMA).
    Por qué: BD T1 define los Sistemas de Información para Ayuda a la
            Toma de Decisiones (DSS). FurniGest ya tiene los datos;
            añadir predicción lo convierte en un DSS completo.
    Coste estimado: 2-3 días.
    En la memoria: nuevo subsección en Cap. 3 Analytics.

────────────────────────────────────────────────────────────────────────
14. HTTPS/TLS EN CAPA DE APLICACIÓN (enforcing en el servidor)
    Asignatura: SI T7 (mecanismos criptográficos: SSL/TLS, AES, certificados),
                Redes T3 (capa de transporte, conexión TCP)
    Qué es: Railway gestiona TLS en el balanceador (nivel de plataforma),
            pero la aplicación no redirige HTTP → HTTPS explícitamente.
            Añadir middleware HTTPS-redirect en FastAPI + HSTS header:
              app.add_middleware(HTTPSRedirectMiddleware)
            También: cabeceras de seguridad (CORS stricto, X-Frame-Options).
    Por qué: SI T7 cubre el protocolo SSL/TLS (AES para el cifrado simétrico,
            RSA para el intercambio de clave) como mecanismo de
            confidencialidad e integridad en la capa de transporte.
            Refleja el pilar CIA implantado en RNF-06.
    Coste estimado: 2-3 horas.
    En la memoria: ampliar RNF-06, nuevo párrafo en sección Despliegue
                   citando TLS y los certificados gestionados por Railway.

────────────────────────────────────────────────────────────────────────
15. HASH DE CONTRASEÑAS CON bcrypt/Argon2 (preparación para auth)
    Asignatura: SI T7 (funciones hash, mecanismos de autenticación)
    Qué es: Cuando se implante la autenticación (#1 de esta lista),
            las contraseñas deben almacenarse como hash irreversible.
            SI T7 explica: SHA-256 para integridad, bcrypt/Argon2 para
            contraseñas (salt + iteraciones para resistir ataques dict.).
            Usar passlib[bcrypt] en FastAPI:
              from passlib.context import CryptContext
              pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")
    Por qué: Almacenar contraseñas en claro es la vulnerabilidad #2 del
            OWASP Top 10 ("Cryptographic Failures"). SI T7 establece que
            los sistemas de autenticación deben usar funciones hash
            con sal para proteger las credenciales.
    Coste estimado: 1 hora (solo configuración, integrar con #1 JWT).
    En la memoria: subsección Seguridad dentro de Backend.
