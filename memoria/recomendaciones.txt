26. DOBLES DE TEST (pytest-mock) EN SERVICIOS EXTERNOS
    Asignatura: AIS T1.2 (pruebas unitarias: test doubles — mock, stub,
                           spy, fake, dummy; estructura Given/When/Then;
                           aislamiento del SUT; principio FIRST)
    Qué es: Los tests actuales de FurniGest llaman directamente a:
              • Groq API (test_analytics.py → endpoint de tendencias)
              • Stripe API (test_stripe.py)
              • Resend/email (test_albaranes.py → albarán con PDF)
            Esto viola el principio de aislamiento de los tests unitarios:
            fallan si la API externa está caída, si la cuota está agotada
            o si las credenciales no están disponibles en CI.

    AIS T1.2 define los tipos de Test Double (Gerard Meszaros, xUnit Test Patterns):
      STUB    — Devuelve valores predeterminados; no verifica llamadas
      MOCK    — Verifica que fue llamado con los parámetros correctos
      SPY     — Como mock pero también ejecuta la implementación real
      FAKE    — Implementación simplificada (e.g. BD SQLite en lugar de PG)
      DUMMY   — Objeto vacío que se pasa pero no se usa en el test

    Estructura Given/When/Then (AIS T1.2):
      GIVEN (precondición): configurar el mock con la respuesta esperada
      WHEN  (acción):       invocar el endpoint o función bajo prueba
      THEN  (verificación): assert sobre la respuesta + assert sobre el mock

    Implementación con pytest-mock (pip install pytest-mock):

      # Mock de la API de Groq (en conftest.py o test_analytics.py)
      def test_tendencias_con_mock(client, mocker):
          # Given
          mock_groq = mocker.patch(
              'backend.app.utils.groq_llm.groq_chat',
              return_value="Tendencia positiva en ventas de Q4"
          )
          # When
          response = client.post("/api/analytics/tendencias", json={...})
          # Then
          assert response.status_code == 200
          assert "Tendencia" in response.json()["respuesta"]
          mock_groq.assert_called_once()   # Verifica que se llamó exactamente 1 vez

      # Mock de Stripe (en test_stripe.py)
      def test_create_checkout_con_mock(client, mocker):
          mocker.patch(
              'stripe.checkout.Session.create',
              return_value=MagicMock(url="https://checkout.stripe.com/test123")
          )
          response = client.post("/api/stripe/create-checkout", json={...})
          assert response.status_code == 200
          assert "checkout.stripe.com" in response.json()["url"]

    Ventajas:
      • Tests independientes de servicios externos (no necesitan API keys)
      • Tests deterministas — siempre devuelven el mismo resultado
      • Tests rápidos — sin latencia de red
      • Tests que pasan en CI sin exponer secrets de producción

    Coste estimado: 2-3 horas (refactorizar tests existentes + nuevos).
    En la memoria: ampliar sección de Pruebas con el concepto de
                   test doubles (AIS T1.2). Tabla Mock/Stub/Spy/Fake.
                   Añadir que los tests de servicios externos usan Mocks
                   para garantizar el principio FIRST (Fast, Isolated...).

24. ESTRATEGIA DE RAMAS Y PULL REQUESTS (Git Branching Models)
    Asignatura: AIS T3.2 (desarrollo colaborativo con Git: modelos de
                           ramas, Pull Requests, code review, integración
                           continua y calidad de commits)
    Qué es: FurniGest realiza commits directamente sobre `main`.
            AIS T3.2 presenta dos modelos principales de branching:
              Gitflow: main + develop + feature/* + release/* + hotfix/*
              • Adecuado para proyectos con releases periódicas
              • develop acumula features; main solo recibe releases estables
              Trunk-Based Development (TBD): main + feature branches efímeras
              • Cada feature branch vive < 1-2 días
              • CI se ejecuta en cada PR antes de mergear a main
              • Compatible con Entrega Continua (AIS T3.6)
            AIS T3.2 también define las 7 reglas de los mensajes de commit
            (Chris Beams, "How to Write a Git Commit Message"):
              1. Separar asunto del cuerpo con una línea en blanco
              2. Limitar el asunto a 50 caracteres
              3. Iniciar el asunto con mayúsculas
              4. No finalizar el asunto con punto
              5. Usar el verbo imperativo en el asunto ("Add", "Fix", no "Added")
              6. No usar líneas mayores de 72 caracteres en el cuerpo
              7. El cuerpo explica QUÉ y POR QUÉ, no cómo (está en el código)

    Implementación recomendada para FurniGest (Trunk-Based, más simple):
      1. Proteger la rama main en GitHub:
           Settings → Branches → Branch protection rule:
           ✅ Require status checks to pass before merging
           ✅ Require at least 1 pull request review before merging
           ✅ Do not allow bypassing the above settings
      2. Crear .github/PULL_REQUEST_TEMPLATE.md:
           ## Descripción del cambio
           ## Tipo: feat / fix / refactor / test / docs / chore
           ## Tests añadidos o actualizados
           ## ¿El pipeline de CI pasa? (✅/❌)
      3. Adoptar Conventional Commits (conventionalcommits.org):
           feat: añadir endpoint de presupuestos
           fix: corregir cálculo de total en albaranes
           test: añadir mocks para API de Groq
           docs: actualizar README con instrucciones de despliegue

    Por qué: El pipeline de CI actual se ejecuta en cada push a main,
            pero no bloquea el merge si falla. Branch protection + PR
            workflow garantiza que ningún commit roto llegue a producción,
            que es el objetivo central de la Integración Continua (AIS T3.3).
    Coste estimado: 1-2 horas (configuración en GitHub + template).
    En la memoria: subsección "Desarrollo colaborativo" en Cap. 3 CI/CD.
                   Comparar Gitflow vs Trunk-Based y justificar la elección.
                   Mostrar ejemplo de mensaje de commit bien formado.

27. SONARCLOUD — QUALITY GATE EN EL PIPELINE DE CI
    Asignatura: AIS P2 (SonarQube: integración con CI, quality gate,
                         cobertura, bugs, code smells, vulnerabilidades)
               AIS T1.4 (calidad del software, métricas objetivas de calidad)
               AIS T1.5 (SCA — análisis estático de código más allá de Ruff)
    Qué es: SonarCloud es SonarQube en la nube, gratuito para repos
            públicos. Analiza el código en cada PR/push y genera un
            informe unificado con:
              • Cobertura de tests (%)
              • Bugs detectados (análisis de flujo de datos)
              • Code smells (mantenibilidad)
              • Vulnerabilidades de seguridad (mapeadas a OWASP/CWE)
              • Duplicación de código (%)
            Define un "quality gate": el CI falla si la cobertura baja
            de un umbral configurable (e.g., 80%) o si hay nuevos bugs
            introducidos en el PR — esto es lo que AIS P2 denomina
            "política de calidad del código" automatizada.

    Implementación en .github/workflows/deploy.yml:

      - name: SonarCloud Scan
        uses: SonarSource/sonarcloud-github-action@master
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
        with:
          args: >
            -Dsonar.projectKey=usuario_furnigest
            -Dsonar.organization=usuario
            -Dsonar.sources=backend,frontend/src
            -Dsonar.python.coverage.reportPaths=coverage.xml
            -Dsonar.javascript.lcov.reportPaths=frontend/coverage/lcov.info

    Configurar sonar-project.properties en la raíz:
      sonar.projectKey=usuario_furnigest
      sonar.organization=usuario
      sonar.sources=backend,frontend/src
      sonar.exclusions=**/node_modules/**,**/__pycache__/**,**/migrations/**
      sonar.python.version=3.12

    Requisito previo: el step de pytest debe generar coverage.xml:
      pytest --cov=backend --cov-report=xml:coverage.xml

    Por qué: Ruff + ESLint (ya en el pipeline) son linters que detectan
            problemas de estilo y errores evidentes. SonarCloud va más
            lejos: detecta bugs semánticos (variables no usadas en todos
            los caminos), vulnerabilidades OWASP y mide la cobertura en
            un cuadro de mando. AIS T1.5 cita SonarQube explícitamente
            como herramienta de SCA, y AIS P2 está dedicada íntegramente
            a su integración en CI. SonarCloud es gratuito para repos
            públicos — ideal para un TFG en GitHub.
    Coste estimado: 2-3 horas (cuenta SonarCloud + configurar acción
                    + resolver los issues iniciales del análisis).
    En la memoria: nueva subsección "Quality Gate con SonarCloud" en
                   Calidad del código o CI/CD. Captura del dashboard con
                   las métricas obtenidas. Citar AIS T1.5 + AIS P2.

18. TRANSACCIONES ACID EN CREACIÓN DE ALBARÁN
    Asignatura: BD T6 (diseño relacional, integridad)
               BD T7 (SQL: BEGIN / COMMIT / ROLLBACK)
               AIS T1.3 (pruebas de integración)
    Problema real: cuando se crea un albarán, el backend realiza
    tres operaciones distintas en secuencia:
      1. INSERT en tabla albaranes + líneas
      2. INSERT en tabla movimientos (registro de fianza)
      3. Enviar email con PDF adjunto
    Si la operación 2 ó 3 falla a mitad, el albarán puede quedar
    creado sin el movimiento económico — inconsistencia de datos.

    Solución: envolver las operaciones 1+2 en una única transacción
    SQLAlchemy (el email queda fuera, ya que es I/O externo):

      with db.begin():
          db.add(albaran)
          db.flush()   # obtener albaran.id
          db.add(movimiento)
      # Solo aquí, con la transacción confirmada, se envía el email
      send_email(...)

    BD T6/T7 definen las propiedades ACID:
      A — Atomicidad: commit o rollback, nunca estado parcial
      C — Consistencia: se respetan todas las restricciones declaradas
      I — Aislamiento: otras sesiones no ven el estado intermedio
      D — Durabilidad: confirmado =&gt; persiste aunque caiga el servidor

    Por qué: Es un bug real del proyecto y la solución demuestra
            conocimiento directo de BD T6/T7. Inúnicamente conexión
            con las propiedades ACID que define la teoría de BD.
    Coste estimado: 2-3 horas (refactorizar api/albaranes.py + 2 tests).
    En la memoria: párrafo en sección Backend explicando las propiedades
                   ACID y cómo se garantizan con el context manager db.begin().

21. WEBSOCKETS PARA NOTIFICACIONES EN TIEMPO REAL
    Asignatura: SD T3 (comunicaciones en sistemas distribuidos:
                       protocolos, sockets, modelos de comunicación)
               Redes T2 (protocolos capa de aplicación, HTTP Upgrade)
    Qué es: Cuando el estado de un albarán cambia (FIANZA → ALMACÉN
            → RUTA → ENTREGADO), el frontend actualmente no se entera
            hasta que el usuario recarga la página.
            Con WebSockets (RFC 6455), el servidor puede emitir el
            evento de cambio a todos los clientes conectados.

    Implementación en FastAPI:
      @app.websocket("/ws")
      async def websocket_endpoint(ws: WebSocket):
          await ws.accept()
          while True:
              # Escucha eventos de cambio de estado desde BD
              msg = await get_next_state_change()
              await ws.send_json({"albaran_id": msg.id,
                                   "estado": msg.estado})

    En el frontend React: useWebSocket hook + invalidar la cache de
    React Query al recibir el evento (0 recarga manual).

    WebSockets vs HTTP:
      HTTP: siempre iniciado por el cliente (pull)
      WebSocket: canal bidireccional full-duplex persistente (push)
      SD T3 cubre este modelo de comunicación orientada a mensajes
      frente al modelo petición-respuesta de HTTP.

    Coste estimado: 1-2 días (endpoint WS + hook React + tests).
    En la memoria: nueva subsección "Comunicación en tiempo real"
                   citando SD T3 y RFC 6455. Diagrama de secuencia.

22. DOCKER MULTI-STAGE BUILD
    Asignatura: SD T4 (contenedores, Docker, optimización de imágenes)
               SO Mód.1 T1 (sistema de ficheros, capas, procesos)
    Qué es: La imagen Docker actual del backend instala todas las
            dependencias de compilación en la imagen final. Con
            multi-stage build se usan dos etapas:
              Stage 1 (builder): instala pip + compila extensiones C
              Stage 2 (runtime): copia solo el .venv, sin tools de build

    Dockerfile multi-stage para el backend:

      # Stage 1 — builder
      FROM python:3.12-slim AS builder
      WORKDIR /app
      COPY requirements.txt .
      RUN pip install --prefix=/install -r requirements.txt

      # Stage 2 — runtime (imagen final)
      FROM python:3.12-slim
      WORKDIR /app
      COPY --from=builder /install /usr/local
      COPY . .
      CMD ["uvicorn", "backend.app.main:app", "--host", "0.0.0.0"]

    Beneficios técnicos:
      • La imagen final no contiene gcc, pip, setuptools ni headers.
      • Reducción de tamaño típica: de ~800 MB a ~150-200 MB.
      • Menor superficie de ataque (SD T4 + SI).
      • Arranque más rápido en Railway al hacer pull de imagen.

    SO Mód.1 relaciona las capas de imagen Docker con el sistema
    de ficheros en capas (UnionFS/OverlayFS), que es esencialmente
    una aplicación del concepto de sistema de ficheros virtual
    estudiado en SO Mód.4 (E/S y sistema de ficheros).
    Coste estimado: 1-2 horas (reescribir Dockerfile + validar build).
    En la memoria: párrafo en subsección Docker sobre multi-stage
                   y tabla comparativa de tamaño antes/después.

23. VERSIONADO SEMÁNTICO (SemVer) + RELEASES EN GITHUB
    Asignatura: AIS T3.5 (artefactos, repositorios y versiones:
                           SemVer, ciclo SNAPSHOT → RELEASE/GA,
                           trazabilidad commit → artefacto)
    Qué es: FurniGest no tiene ninguna estrategia de versiones.
            La imagen Docker solo tiene la etiqueta `:latest`, que no
            permite saber qué versión está en producción ni hacer rollback.
            AIS T3.5 define Semantic Versioning (SemVer — semver.org):
              MAJOR.MINOR.PATCH — ejemplo: 1.0.0
              • MAJOR: cambios incompatibles con versiones anteriores
              • MINOR: nueva funcionalidad compatible hacia atrás
              • PATCH: corrección de errores compatible hacia atrás
            También define el ciclo de vida del artefacto:
              BUILD/SNAPSHOT → ALPHA → BETA → RC (Release Candidate)
                             → RELEASE / GA (General Availability)
            Versiones pre-release en SemVer: X.Y.Z-identificador
              Ejemplos: 1.0.0-alpha, 1.0.0-beta.1, 1.0.0-rc.1
              Precedencia: 1.0.0-alpha < 1.0.0-beta < 1.0.0-rc.1 < 1.0.0

    Implementación en FurniGest:
      1. Crear tag inicial en git:
           git tag -a v1.0.0 -m "Primera versión estable de FurniGest"
           git push origin v1.0.0
      2. Crear GitHub Release (desde GitHub.com) con changelog manual.
      3. Publicar imagen Docker versionada en GitHub Actions:
           - name: Build and push Docker image
             uses: docker/build-push-action@v5
             with:
               tags: |
                 ghcr.io/usuario/furnigest:latest
                 ghcr.io/usuario/furnigest:${{ github.ref_name }}
      4. Actualizar docker-compose.yml para usar imagen versionada:
           image: ghcr.io/usuario/furnigest:1.0.0

    Por qué: AIS T3.5 define la trazabilidad commit → artefacto como
            requisito fundamental del proceso de CI/CD. Sin versiones, no
            es posible hacer rollback, auditar qué hay en producción ni
            comparar builds. Para el TFG, un GitHub Release con tag v1.0.0
            le da al evaluador un punto de referencia formal y reproducible.
    Coste estimado: 1-2 horas (git tag + GitHub Release + ajuste CI).
    En la memoria: párrafo en sección CI/CD sobre estrategia de versiones.
                   Tabla con ciclo de vida SNAPSHOT → RELEASE/GA y ejemplos
                   de versiones pre-release. Citar semver.org.

25. TIPOS DE MANTENIMIENTO IEEE EN LA MEMORIA (solo texto)
    Asignatura: AIS T2.1 (mantenimiento del software: IEEE 1219, tipos
                           de mantenimiento, leyes de Lehman y Belady,
                           proyectos vs productos, retos del mantenimiento)
    Qué es: IEEE define formalmente cuatro tipos de mantenimiento.
            FurniGest ha pasado exactamente por todos ellos durante su
            desarrollo iterativo — solo hay que nombrarlo en la memoria.

    Los cuatro tipos IEEE de mantenimiento:
      CORRECTIVO — Reparación de defectos y fallos detectados en producción
      PERFECTIVO — Mejoras de funcionalidad o rendimiento sin corregir bugs
      ADAPTATIVO — Adaptación a cambios en el entorno (nueva versión de
                   librería, cambio en API externa, migración de plataforma)
      PREVENTIVO — Restructuración del código para facilitar futuros
                   cambios (refactorización, mejora de documentación,
                   añadir tests que antes no existían)

    Conexión con el historial de commits de FurniGest:
      ✓ Correctivo: commits "fix: ..." (errores en endpoints, cálculos)
      ✓ Perfectivo: commits "feat: ..." (añadir analíticas, módulo IA)
      ✓ Adaptativo: actualizaciones de FastAPI, SQLAlchemy, Vite, React
      ✓ Preventivo: refactorizaciones, type hints, tests a posteriori

    Leyes de Lehman y Belady (AIS T2.1):
      • Ley del cambio continuo: un software usado en un entorno real
        debe cambiar; de lo contrario se vuelve progresivamente inútil.
      • Ley del crecimiento continuo: la funcionalidad debe aumentar
        continuamente para mantener la satisfacción de los usuarios.

    FurniGest como "producto" (no "proyecto"):
      AIS T2.1 distingue entre proyectos (con fin definido, 20-30% del
      coste total es mantenimiento) y productos (en evolución constante,
      60-80% del coste total es post-release). El TFG entregado que sigue
      creciendo encaja en el modelo de "producto" con MVP + iteraciones,
      y el propio desarrollo ágil se considera mantenimiento continuo.

    Coste estimado: 0 código. 1-2 horas redactando en Cap. 2.
    En la memoria: añadir subsección "Mantenimiento del software" en
                   Cap. 2 Metodología. Tabla de 4 tipos con ejemplos reales
                   del historial de commits de FurniGest. Mencionar las
                   leyes de Lehman como justificación del diseño evolutivo.

29. ENDPOINT /health + READINESS PROBE EN RAILWAY
    Asignatura: SD T1 (detección y recuperación de fallos en SSDD,
                        disponibilidad, monitorización de servicios)
               AIS T3.4 (pipelines de CI/CD, despliegue)
               SD T4 (Docker, orquestación, contenedores en producción)
    Qué es: FurniGest no expone ningún endpoint de salud.
            Un health check es un endpoint estándar de la industria que
            informa si el servicio está listo para atender peticiones:

      @app.get("/health")
      async def health_check(db: Session = Depends(get_db)):
          try:
              db.execute(text("SELECT 1"))      # Verifica BD activa
              return {"status": "ok", "db": "connected"}
          except Exception as e:
              raise HTTPException(status_code=503, detail=str(e))

    Railway puede usar este endpoint para:
      - Reiniciar automáticamente la instancia si devuelve 503
      - No enviar tráfico hasta que el servicio confirme readiness
      - Garantizar zero-downtime deploys (sustituye contenedor viejo
        solo cuando el nuevo responde 200 en /health)

    Conexión académica:
      SD T1 define la disponibilidad como propiedad fundamental de los
      sistemas distribuidos y establece que "los fallos deben detectarse
      y recuperarse automáticamente". El health check es la implementación
      operacional de ese principio: el orquestador (Railway / Kubernetes)
      detecta el fallo consultando periódicamente el endpoint y actúa
      sin intervención humana.
      En Kubernetes se llama "liveness probe" (¿está vivo?) y
      "readiness probe" (¿está listo para recibir tráfico?) — conceptos
      que extienden naturalmente SD T4 (contenedores) hacia orquestación.

    Coste estimado: 30 minutos (10 líneas de código + config Railway).
    En la memoria: párrafo en subsección Despliegue: "el servicio expone
                   /health conforme al principio de detección automática
                   de fallos de los sistemas distribuidos (SD T1)".

30. PAGINACIÓN EN ENDPOINTS DE LISTA
    Asignatura: BD T6 (diseño físico, optimización de consultas)
               SD T1 (escalabilidad en sistemas distribuidos)
               IS T2 (RNF de rendimiento)
    Qué es: Todos los endpoints GET del proyecto (albaranes, clientes,
            productos, movimientos) devuelven TODOS los registros en una
            sola respuesta. Con 10.000 albaranes esto provoca:
              - Consulta SQL sin LIMIT → seq scan completo O(n)
              - Payload JSON de varios MB innecesario
              - El frontend renderiza una tabla enorme → UI bloqueada

    Hay dos patrones estándar de paginación:

      Offset pagination (más simple):
        GET /api/albaranes?page=1&limit=20&sort=fecha&order=desc
        Response: { "items": [...], "total": 843, "page": 1, "pages": 43 }

        En FastAPI:
          @router.get("/albaranes")
          def get_albaranes(page: int = 1, limit: int = 20,
                            db: Session = Depends(get_db)):
              offset = (page - 1) * limit
              total  = db.query(AlbaranDB).count()
              items  = db.query(AlbaranDB).offset(offset).limit(limit).all()
              return {"items": items, "total": total, "page": page,
                      "pages": math.ceil(total / limit)}

      Cursor pagination (más eficiente para grandes volúmenes):
        Usa el id o fecha del último registro como cursor:
        GET /api/albaranes?cursor=2024-03-15T10:00:00&limit=20
        No necesita COUNT(*) ni calcula offsets → O(log n) con índice

    Por qué:
      BD T6 establece que el diseño físico debe optimizar los planes de
      ejecución de las consultas más frecuentes. Una query sin LIMIT obliga
      al motor a recuperar y transmitir todos los registros aunque el usuario
      solo vea 20. Añadir LIMIT/OFFSET + el índice de fecha/estado (#19)
      convierte un seq scan O(n) en un index range scan O(log n + k).
      SD T1 define la escalabilidad como la capacidad del sistema de
      soportar mayor carga sin degradación: sin paginación, el backend
      escala mal aunque se tenga un clúster de contenedores.
    Coste estimado: 3-4 horas (refactorizar endpoints + actualizar
                    la tabla del frontend con controles de paginación).
    En la memoria: párrafo en sección Backend sobre paginación.
                   Comparativa offset vs cursor con complejidades.
                   Mencionar como mejora de RNF-05 (rendimiento).

32. TAREAS EN SEGUNDO PLANO (FastAPI BackgroundTasks / Celery)
    Asignatura: SO T2/T3 (procesos, hilos, concurrencia, scheduling)
               SD T3 (comunicación asíncrona, colas de mensajes)
               AIS T1 (pruebas de rendimiento, no-functional reqs)
    Qué es: FurniGest envía el email con el PDF del albarán de forma
            síncrona: el cliente HTTP espera hasta que el email se envía
            antes de recibir la respuesta 201. Si el servidor de email
            (Resend) tarda 2 segundos, la operación de crear un albarán
            tarda 2 segundos de más.

    El patrón correcto es ejecutar el envío de email en segundo plano:

      Opción A — FastAPI BackgroundTasks (sin dependencias extra):
        from fastapi import BackgroundTasks

        @router.post("/albaranes")
        async def create_albaran(data: AlbaranCreate,
                                  background_tasks: BackgroundTasks,
                                  db: Session = Depends(get_db)):
            albaran = create_albaran_in_db(db, data)
            # Respuesta inmediata al cliente
            background_tasks.add_task(send_albaran_email, albaran)
            return albaran   # 201 devuelto ANTES de que se envíe el email

      Opción B — Celery + Redis (para proyectos con mayor escala):
        La tarea de email se encola en Redis y la ejecuta un worker Celery
        en un proceso separado. Permite reintentos automáticos si falla.
        Relacionado con SD T3 (colas de mensajes, modelo productor/consumidor)
        y SO T3 (procesos ligeros, comunicación entre procesos via IPC/queue).

    Conexión académica:
      SO T2 define el modelo de procesos: un proceso puede delegar trabajo
      a otro proceso o hilo para continuar su ejecución sin bloquearse.
      FastAPI BackgroundTasks es la implementación de este paradigma en
      ASGI: el event loop despacha la tarea de I/O al background mientras
      el handler principal libera el hilo al pool (SO T3: scheduling
      cooperativo). SD T3 generaliza esto con colas de mensajes distribuidas
      (modelo productor/consumidor, RabbitMQ, Celery).

    Coste estimado: 1-2 horas (refactorizar api/albaranes.py para usar
                    BackgroundTasks + adaptar tests con mock del background).
    En la memoria: párrafo en sección Backend sobre "ejecución asíncrona
                   de tareas I/O" citando SO T2/T3 y SD T3. Diagrama de
                   secuencia comparando síncrono vs asíncrono.

33. PRUEBAS DE CARGA CON LOCUST
    Asignatura: IS T2 (RNF de rendimiento: tiempo de respuesta, throughput)
               AIS T1 (tipos de pruebas: funcionales vs no funcionales)
               SD T1 (escalabilidad, comportamiento bajo carga)
    Qué es: Las pruebas actuales de FurniGest son todas funcionales
            (verifican si la lógica es correcta, no si es rápida).
            Las pruebas de carga (load testing) verifican los RNF de
            rendimiento: "el sistema debe responder en < 500ms con 50
            usuarios concurrentes" (IS T2 — requisitos no funcionales).

    Locust es una herramienta de load testing escrita en Python
    (https://locust.io) donde el script de prueba es código Python puro:

      # locustfile.py
      from locust import HttpUser, task, between

      class FurniGestUser(HttpUser):
          wait_time = between(1, 3)   # Espera aleatoria 1-3s entre tasks

          @task(3)                    # Peso 3: tarea más frecuente
          def listar_albaranes(self):
              self.client.get("/api/albaranes")

          @task(1)                    # Peso 1: tarea menos frecuente
          def analytics_summary(self):
              self.client.get("/api/analytics/summary")

    Ejecutar: locust --headless -u 50 -r 5 --run-time 1m
              (50 usuarios, 5 nuevos por segundo, durante 1 minuto)

    Métricas que genera Locust:
      • Requests/second (throughput)
      • Average/median/p95/p99 response time
      • Failure rate (% de peticiones con error)
      • Gráfico de evolución temporal de latencia

    Conexión académica:
      IS T2 define los RNF de rendimiento en términos de tiempo de
      respuesta máximo y throughput mínimo. Sin pruebas de carga, estos
      RNF son requisitos no verificados — Locust los convierte en
      pruebas ejecutables y reproducibles, igual que pytest convierte
      los RF en tests automatizados.
      AIS T1 incluye explícitamente las pruebas no funcionales
      (rendimiento, seguridad, escalabilidad) dentro de la pirámide de
      pruebas en su nivel más alto (pruebas de sistema).
      SD T1 define la escalabilidad: con Locust se puede medir si añadir
      una réplica del backend en Railway realmente duplica el throughput.

    Coste estimado: 2-3 horas (instalar Locust + escribir locustfile.py
                    + interpretar y documentar los resultados).
    En la memoria: subsección "Pruebas de rendimiento" en Cap. 3/4.
                   Tabla con throughput y p95 latency bajo 50 usuarios.
                   Screenshot del dashboard de Locust. Relacionar con RNF-05.
