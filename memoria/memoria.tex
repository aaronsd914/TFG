\chapter*{Resumen}

El presente Trabajo Fin de Grado describe el diseño e implementación de \textbf{FurniGest}, una aplicación web de gestión integral (ERP, \textit{Enterprise Resource Planning}) orientada a una tienda de muebles. La problemática de partida es la dependencia de métodos manuales ---hojas de cálculo y registros en papel--- para dar soporte a los procesos de negocio fundamentales: gestión de clientes, pedidos, inventario, flujos financieros y logística de entrega. Esta aproximación genera ineficiencias operativas, propensión a errores y ausencia total de visibilidad analítica sobre el rendimiento del negocio.

La solución desarrollada es una \textit{Single Page Application} (SPA) con arquitectura cliente-servidor desacoplada. El \textit{backend} está construido con \textbf{FastAPI} (Python 3.12) y \textbf{SQLAlchemy} sobre una base de datos \textbf{PostgreSQL~16}, exponiendo una API REST que gestiona ocho entidades de dominio. El \textit{frontend} emplea \textbf{React~19}, \textbf{Vite~6} y \textbf{TailwindCSS~4}, con visualización de datos mediante \textbf{Chart.js}.

Entre las funcionalidades avanzadas destaca un motor de analíticas con segmentación de clientes mediante el modelo \textbf{RFM} (\textit{Recency-Frequency-Monetary}) y análisis de pares de productos frecuentemente comprados juntos (\textit{market basket analysis}). Se integra además un asistente de inteligencia artificial basado en la API de \textbf{Groq} con el modelo \textbf{Llama-3.1-8b-instant}, que recibe métricas de negocio en tiempo real y responde consultas en lenguaje natural. El sistema incorpora también integración de pagos con \textbf{Stripe Checkout}, generación de documentos PDF con \textbf{ReportLab} y notificaciones por correo electrónico mediante SMTP y la API de \textbf{Resend}.

La aplicación se despliega en producción con el \textit{frontend} en \textbf{Vercel} y el \textit{backend} en \textbf{Railway}, con un flujo de integración y despliegue continuos (\textit{CI/CD}) articulado sobre \textbf{GitHub Actions}. La batería de pruebas automatizadas comprende 96 pruebas unitarias del \textit{backend} (pytest) y 61 del \textit{frontend} (Vitest), con cobertura de todos los módulos de la API.

La digitalización de los procesos de negocio representa uno de los mayores retos para las pequeñas y medianas empresas del sector minorista. En este contexto, el presente Trabajo Fin de Grado aborda el diseño y la implementación de \textbf{FurniGest}, una aplicación web de gestión integral orientada a una tienda de muebles, con el objetivo de sustituir los métodos manuales de trabajo por una herramienta digital centralizada, accesible y extensible. FurniGest puede clasificarse como un \textbf{Sistema de Información para la Gestión (SIG)}: un sistema que captura, almacena, procesa y distribuye información para apoyar la toma de decisiones empresariales.

\section{Contexto y alcance}

Las tiendas de muebles combinan de forma habitual operaciones propias de la venta al detalle ---gestión de clientes, catálogo de productos y presupuestos--- con procesos logísticos que distinguen este sector de otros: cada pedido implica la preparación de un albarán, el seguimiento del estado del producto en almacén y, finalmente, la entrega a domicilio mediante un servicio de transporte propio. El control financiero de estos flujos ---ingresos por ventas, fianzas, gastos de transporte y devoluciones--- es igualmente complejo.

En el caso de partida que motiva este proyecto, la tienda gestionaba todos estos procesos de forma manual, haciendo uso de hojas de cálculo y registros en papel. Esta aproximación presentaba limitaciones evidentes: duplicación de datos, ausencia de trazabilidad sobre el estado de los pedidos, imposibilidad de obtener analíticas de ventas en tiempo real y dificultad para generar documentación formal (albaranes en PDF) de forma ágil.

FurniGest cubre íntegramente estas necesidades mediante una \textit{Single Page Application} (SPA) con una API REST en el \textit{backend}. El alcance del sistema comprende:

\begin{enumerate}
  \item Gestión completa (CRUD) de clientes, productos y proveedores.
  \item Ciclo de vida de pedidos (albaranes) con una máquina de cuatro estados: \textit{Fianza} $\to$ \textit{Almacén} $\to$ \textit{Ruta} $\to$ \textit{Entregado}.
  \item Libro contable de movimientos financieros con creación automática de registros ante eventos de negocio.
  \item Dashboard con indicadores clave de rendimiento (KPI) y gráficas interactivas.
  \item Motor de analíticas avanzadas: segmentación RFM de clientes y análisis de cestas de la compra.
  \item Asistente de inteligencia artificial en lenguaje natural, integrado con métricas de negocio en vivo.
  \item Módulo de logística y transporte con gestión de camiones y liquidación de rutas.
  \item Integración de pagos en línea mediante Stripe Checkout.
  \item Generación de documentos PDF (albaranes, facturas de ruta e informes de analíticas).
  \item Notificaciones por correo electrónico con PDF adjunto al crear cada pedido.
\end{enumerate}

La aplicación está desplegada en producción, con el \textit{frontend} en Vercel y el \textit{backend} en Railway. La base de datos es PostgreSQL~16. Todo el proceso de integración y despliegue continuos está automatizado mediante GitHub Actions.

\chapter{Objetivos}

Este capítulo describe el problema que motiva el desarrollo de FurniGest, analiza las alternativas existentes en el mercado y expone la metodología de trabajo seguida a lo largo del proyecto.

\section{Descripción del problema}

La tienda de muebles que sirve de caso de uso contaba, antes del inicio de este proyecto, con el siguiente modelo operativo:

\begin{itemize}
  \item Los datos de clientes se almacenaban en una hoja de cálculo local sin mecanismo de búsqueda estructurado.
  \item Los pedidos se registraban manualmente en papel, sin trazabilidad del estado (si el artículo estaba en almacén, en ruta o entregado).
  \item La facturación y los albaranes se generaban con plantillas de procesador de texto, con el riesgo de errores tipográficos y pérdida de documentos.
  \item No existía ninguna herramienta que ofreciese visión consolidada de los ingresos, gastos o tendencias de venta.
  \item El cobro de fianzas y el seguimiento de los pagos pendientes se realizaban de memoria o con notas en papel.
\end{itemize}

Esta situación generaba ineficiencias tangibles: tiempo perdido en tareas administrativas repetitivas, errores en los importes reflejados en los albaranes, falta de coordinación entre el personal de tienda y los repartidores, y ausencia total de capacidad analítica para la toma de decisiones.

El objetivo global del proyecto es diseñar e implementar un sistema de información web que digitalice de forma integral todos estos procesos. El sistema debe ser accesible desde cualquier ordenador de la red local de la tienda sin necesidad de instalar ningún software adicional, y debe poder operar también de forma remota a través de internet una vez desplegado en la nube.

\section{Estudio de alternativas}

Antes de optar por el desarrollo a medida se analizaron las soluciones de gestión comercial disponibles en el mercado:

\begin{table}[H]

  \caption{Comparativa de soluciones ERP existentes.}

Las soluciones comerciales presentan un problema común: están diseñadas para cubrir una amplísima gama de sectores y tamaños de empresa, lo que se traduce en interfaces sobrecargadas y flujos de trabajo que no encajan con la operativa específica de una tienda de muebles mediana. Ninguna de las alternativas evaluadas ofrecía de forma nativa la combinación de módulo logístico por camiones, segmentación RFM de clientes y asistente conversacional de IA.

Por estos motivos se optó por el \textbf{desarrollo a medida} de una aplicación web moderna, aprovechando tecnologías de código abierto y servicios en la nube con capa gratuita amplia (Railway, Vercel, Groq, Stripe en modo de prueba).

\section{Metodología empleada}

El desarrollo de FurniGest siguió el \textbf{modelo iterativo e incremental} \cite{pressman}, inspirado en los principios del \textbf{Manifiesto Ágil} \cite{manifiestoAgil} (Beck et al., 2001) y en las prácticas de gestión de \textbf{Scrum} \cite{schwaber}. Frente al modelo en cascada clásico ---en el que los requisitos se congelan antes del diseño--- el enfoque ágil permite ajustar el \textit{product backlog} a medida que se adquiere un mejor conocimiento del dominio del problema.

Cada iteración del proyecto agrupaba un conjunto de historias de usuario que añadían valor entrenable de forma independiente, siguiendo el ciclo: análisis de requisitos de la iteración $\to$ diseño $\to$ implementación $\to$ pruebas $\to$ revisión. La sincronización entre el estado del código y las pruebas automatizadas se mantuvo gracias al pipeline de CI que se activaba en cada \textit{push}.

Las fases del desarrollo fueron:

\begin{enumerate}
  \item \textbf{Fase~0 —~Elicitación y análisis de requisitos:} Se identificaron las entidades del dominio y los casos de uso principales mediante entrevistas con la tienda. Se elaboró el catálogo de requisitos funcionales (RF) y no funcionales (RNF) siguiendo la estructura del estándar\textbf{ IEEE~830} \cite{ieee830}. Las funcionalidades se priorizaron en dos categorías: \textit{core} (necesarias para la operativa diaria) y \textit{avanzadas} (analíticas, IA, pagos en línea).
  \item \textbf{Fase~1 —~Implementación iterativa por módulos:} Desarrollo incremental guiado por pruebas (\textbf{TDD}, \textit{Test-Driven Development} \cite{beckTDD}). En cada iteración se completaban: modelo de datos, endpoints de la API, vista del \textit{frontend} y pruebas automatizadas. El orden de implementación respetó las dependencias entre entidades (clientes $\to$ productos/proveedores $\to$ albaranes $\to$ movimientos $\to$ módulos avanzados).
  \item \textbf{Fase~2 —~Integración y entrega continuas:} Desde la primera semana se configuró un pipeline de GitHub Actions con \textbf{integración continua (CI)} \cite{fowlerCI} que ejecuta la batería de pruebas en cada \textit{push} y despliega automáticamente en producción \textbf{(despliegue continuo, CD)} si todas las pruebas pasan.
  \item \textbf{Fase~3 —~Refinamiento e inspección de calidad:} Una vez implementadas las funcionalidades base se realizó una pasada de \textbf{refactorización} \cite{fowlerRefactoring} para reducir la \textit{deuda técnica} acumulada, mejorar la cohesión de módulos y eliminar duplicaciones de código. Se aplicó la regla del \textit{boy scout}: ``deja el código mejor de lo que te lo encontraste''.
\end{enumerate}

\section{Diseño}

\subsection{Arquitectura del sistema}

FurniGest puede caracterizarse como un \textbf{sistema distribuido} en el sentido definido por Tanenbaum: un conjunto de nodos independientes (cliente en Vercel, servidor en Railway, base de datos en PostgreSQL) que se presentan al usuario como un único sistema coherente~\cite{tanenbaum}. Sus características fundamentales son: \textbf{concurrencia} (múltiples usuarios pueden operar simultáneamente gracias al modelo ASGI de FastAPI), \textbf{tolerancia a fallos} (cada servicio de Railway y Vercel está respaldado por infraestructura con disponibilidad declarada $\geq$ 99\,\%) y \textbf{escalabilidad horizontal} (añadir instancias del backend sin modificar la capa de datos).

Desde la perspectiva de las garantías de consistencia, la elección de PostgreSQL se encuadra en el \textbf{teorema CAP} (Brewer, 2000)~\cite{brewer2000}: en un sistema distribuido sólo es posible garantizar simultáneamente dos de las tres propiedades —\textbf{consistencia} (\textit{Consistency}), \textbf{disponibilidad} (\textit{Availability}) y \textbf{tolerancia a particiones} (\textit{Partition tolerance})—. PostgreSQL prioriza la consistencia y la tolerancia a particiones (sistema \textbf{CP}), sacrificando la disponibilidad total durante una partición de red. Para FurniGest, este compromiso es el adecuado: la integridad de los datos financieros y de inventario tiene mayor prioridad que la disponibilidad ante fallos de red infrecuentes.

FurniGest sigue una \textbf{arquitectura de tres capas} \cite{pressman}, también representable mediante un \textit{diagrama de despliegue} UML: capa de presentación (cliente), capa lógica (servidor) y capa de datos (base de datos). Este patrón arquitectónico \textbf{desacopla} las responsabilidades de cada nivel, favoreciendo la \textbf{cohesión} interna de cada subsistema y reduciendo el \textbf{acoplamiento} entre ellos. La comunicación entre el \textit{frontend} y el \textit{backend} se realiza íntegramente mediante peticiones HTTP (REST), lo que permite que ambas capas evolucionen de forma independiente y que el \textit{backend} pueda ser consumido por cualquier otro cliente en el futuro (p.ej., aplicación móvil).

\subsection{Modelo de datos}

El diseño de la base de datos sigue la metodología clásica de modelado conceptual mediante el \textbf{modelo Entidad-Relación (E/R)}, propuesto por Peter P.~Chen en 1976~\cite{chen1976}. En el nivel conceptual se identifican las entidades del dominio, sus atributos y las relaciones entre ellas. Posteriormente, el modelo E/R se transforma al \textbf{modelo relacional} de E.F.~Codd~\cite{codd1970}: cada entidad se convierte en una tabla, las relaciones pasan a ser \textbf{claves foráneas} (\textit{foreign keys}) y se aplican \textbf{restricciones de integridad referencial} en el SGBD para garantizar la consistencia de los datos. SQLAlchemy actúa como \textbf{ORM} (\textit{Object-Relational Mapper}), mapeando las clases Python a tablas SQL y abstrayendo el acceso a datos en una interfaz orientada a objetos.

El esquema de la base de datos cumple la \textbf{tercera forma normal (3FN)}: cada atributo no clave depende únicamente de la clave primaria de su tabla y no de otros atributos no clave. Esta propiedad elimina redundancias y evita las anomalías de actualización, inserción y borrado que surgen en esquemas sin normalizar. Por ejemplo, los datos del proveedor no se repiten en cada fila de la tabla de productos, sino que se referencian mediante la clave foránea \texttt{proveedor\_id}; del mismo modo, el precio de cada línea de albarán se almacena de forma independiente al precio de catálogo del producto, preservando el histórico de ventas sin acoplar ambas tablas.


\subsection{Diseño de la API REST}

La API REST de FurniGest sigue el estilo arquitectónico \textbf{REST} (\textit{Representational State Transfer}), descrito por Roy Fielding en su tesis doctoral en el año 2000~\cite{fieldingREST}. REST define un conjunto de restricciones para el diseño de servicios web distribuidos: sin estado (\textit{stateless}), interfaz uniforme basada en \textbf{recursos} identificados por \textbf{URIs}, y uso de los métodos estándar del protocolo HTTP (GET, POST, PUT, PATCH, DELETE). El carácter \textit{stateless} implica que cada petición del cliente contiene toda la información necesaria para ser procesada, sin que el servidor mantenga estado de sesión entre peticiones. Esta restricción facilita la escalabilidad horizontal del servidor: dado que no existe estado compartido entre instancias, es posible añadir nodos al servicio sin coordinación entre ellos.

La API REST se sirve en \texttt{http://\{host\}:8000}. Todos los endpoints de dominio llevan el prefijo \texttt{/api}. El protocolo de transporte utilizado es \textbf{HTTP/1.1} (RFC~2616)~\cite{kurose}, el estándar de la capa de aplicación sobre el que se construye REST. En producción, Railway inyecta \textbf{TLS} (\textit{Transport Layer Security}) de forma automática, de modo que toda la comunicación entre el cliente y el servidor discurre cifrada mediante HTTPS. \subsection{Backend con FastAPI}
\label{sec:backend}

El \textit{backend} es una aplicación FastAPI~0.115 \cite{fastapi} que se arranca con Uvicorn~\cite{uvicorn}. Su punto de entrada es \texttt{backend/app/main.py}, que registra todos los routers, configura el \textit{middleware} CORS y define el ciclo de vida de la aplicación (\textit{lifespan}) en el que se ejecutan la creación de tablas y la carga de datos iniciales.

Uvicorn implementa el protocolo \textbf{ASGI} (\textit{Asynchronous Server Gateway Interface}), que permite gestionar múltiples peticiones HTTP concurrentes en un único proceso mediante el \textbf{bucle de eventos} (\textit{event loop}) de Python. A diferencia del modelo clásico basado en un hilo del sistema operativo por petición, el modelo asíncrono suspende una corrutina mientras espera I/O (base de datos, red) y cede el control a otra petición pendiente, dando lugar a una alta concurrencia sin el coste de la creación y conmutación de hilos.

El Código~\ref{cod:lifespan} muestra el patrón de inicialización empleado.

\begin{mypython}[float={!t},caption={Ciclo de vida de la aplicación FastAPI (\texttt{main.py}).},label={cod:lifespan}]
from contextlib import asynccontextmanager
from fastapi import FastAPI
from backend.app.database import Base, engine, SessionLocal
from backend.app.seed import seed

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Crea todas las tablas si no existen
    Base.metadata.create_all(bind=engine)
    # Carga datos de demostración en primera ejecución
    with SessionLocal() as db:
        seed(db)
    yield  # la aplicación queda en ejecución

app = FastAPI(lifespan=lifespan)
\end{mypython}

La gestión de sesiones de base de datos sigue el patrón \textit{dependency injection} de FastAPI \cite{sqlalchemy}. El generador \texttt{get\_db()} garantiza que cada petición HTTP dispone de una sesión exclusiva que se cierra automáticamente aunque ocurra una excepción:

\begin{mypython}[float={!t},caption={Generador de sesión de base de datos (\texttt{database.py}).},label={cod:getdb}]
from sqlalchemy.orm import Session

def get_db():
    db: Session = SessionLocal()
    try:
        yield db
    finally:
        db.close()
\end{mypython}

Los routers inyectan esta sesión declarando \texttt{db: Session = Depends(get\_db)} en la firma de cada función de vista. Pydantic~v2 \cite{pydantic} se usa para validar la entrada de cada petición (modelos \texttt{*Create}) y serializar la respuesta (modelos de respuesta con \texttt{from\_attributes=True}).

\paragraph{Test-Driven Development.}
El desarrollo del \textit{backend} siguió el ciclo \textbf{TDD} (\textit{Test-Driven Development}) descrito por Kent Beck \cite{beckTDD}: \textbf{Rojo} (escribir el test que falla) $\to$ \textbf{Verde} (escribir el código mínimo para que pase) $\to$ \textbf{Refactor} (mejorar la estructura sin alterar el comportamiento). Este ciclo garantiza que cada endpoint de la API tiene al menos un test que lo especifica antes de su implementación, actuando los tests como \textit{especificación ejecutable} del sistema.

\subsubsection{Segmentación RFM}

El modelo \textit{Recency-Frequency-Monetary} \cite{hughes1994, rfm_review} clasifica a los clientes en función de tres dimensiones calculadas sobre la tabla \texttt{albaranes}:

\begin{itemize}
  \item \textbf{Recencia ($R$):} días transcurridos desde el último albarán (a menor número, mejor).
  \item \textbf{Frecuencia ($F$):} número total de albaranes del cliente.
  \item \textbf{Monetario ($M$):} suma del campo \texttt{total} de todos sus albaranes.
\end{itemize}

Cada dimensión se cuantifica en una escala del 1 al 5 mediante quintiles calculados sobre el conjunto de todos los clientes activos. La combinación de las tres puntuaciones determina el segmento, según las reglas de la Tabla~\ref{tab:rfm_segmentos}. El Algoritmo~\ref{alg:rfm} muestra el pseudocódigo del proceso.

\begin{table}[H]
  \centering
  \caption{Reglas de asignación de segmentos RFM.}
  \label{tab:rfm_segmentos}
  \renewcommand{\arraystretch}{1.4}
  \begin{footnotesize}
  \begin{tabular}{|l|p{7cm}|}
    \hline
    \textbf{Segmento} & \textbf{Criterio (puntuaciones R, F, M sobre quintil 1--5)} \\
    \hline
    Champions          & $r = 5$ y $f \geq 4$ \\
    Loyal Customers    & $f \geq 3$ y $m \geq 3$ \\
    Potential Loyalists & $r \geq 3$ y $f \leq 2$ \\
    Need Attention     & $r = 3$ y $f \leq 2$ y $m \leq 2$ \\
    At Risk            & $r \leq 2$ y $f \geq 2$ \\
    Lost               & $r = 1$ y $f = 1$ \\
    \hline
  \end{tabular}
  \end{footnotesize}
\end{table}

\begin{algorithm}[H]
\begin{algorithmic}[1]
\STATE Consultar todos los clientes con al menos un albarán
\FORALL{cliente $c$ en el conjunto}
  \STATE $R_c \leftarrow$ días entre hoy y la fecha del último albarán de $c$
  \STATE $F_c \leftarrow$ número total de albaranes de $c$
  \STATE $M_c \leftarrow \sum \texttt{total}$ de todos los albaranes de $c$
\ENDFOR
\STATE Calcular quintiles sobre las distribuciones $\{R_c\}$, $\{F_c\}$, $\{M_c\}$
\FORALL{cliente $c$}
  \STATE $r_c \leftarrow$ posición de $R_c$ en el quintil de recencia (invertido: menor $R$ $\Rightarrow$ mayor puntuación)
  \STATE $f_c \leftarrow$ posición de $F_c$ en el quintil de frecuencia
  \STATE $m_c \leftarrow$ posición de $M_c$ en el quintil de monetario
  \STATE $\textit{segmento}_c \leftarrow$ \textsc{AsignarSegmento}$(r_c, f_c, m_c)$
\ENDFOR
\RETURN mapa $c \mapsto \textit{segmento}_c$
\end{algorithmic}
\caption{Segmentación RFM de clientes.}
\label{alg:rfm}
\end{algorithm}


\subsubsection{Análisis de cestas de la compra}

El análisis de \textit{market basket} \cite{agrawal1994} identifica pares de productos que aparecen con frecuencia en el mismo albarán. El algoritmo recorre todos los albaranes y cuenta las co-ocurrencias de cada par ordenado $(p_i, p_j)$ con $i < j$. Los pares cuya frecuencia supera un umbral mínimo (\textit{min\_support}) se devuelven ordenados por frecuencia descendente, ofreciendo información accionable para estrategias de venta cruzada. El Algoritmo~\ref{alg:basket} detalla el proceso.

\begin{algorithm}[H]
\begin{algorithmic}[1]
\STATE $\textit{pares} \leftarrow \{\}$ \COMMENT{mapa par $\to$ contador}
\FORALL{albarán $a$ en la tabla \texttt{albaranes}}
  \STATE $P_a \leftarrow$ lista de \texttt{producto\_id} en las líneas de $a$
  \FORALL{par $(p_i, p_j)$ con $p_i < p_j$ y $p_i, p_j \in P_a$}
    \STATE $\textit{pares}[(p_i, p_j)] \mathrel{+}= 1$
  \ENDFOR
\ENDFOR
\STATE Filtrar $\textit{pares}$ conservando aquéllos con valor $\geq \textit{min\_support}$
\STATE Ordenar por valor descendente
\RETURN primeros $k$ pares
\end{algorithmic}
\caption{Análisis de pares de productos en cestas de la compra.}
\label{alg:basket}
\end{algorithm}


\subsection{Asistente de inteligencia artificial}
\label{sec:ia}

El asistente de IA está implementado en \texttt{backend/app/api/ai.py} y se apoya en la API de \textbf{Groq} \cite{groq} con el modelo \textbf{Llama-3.1-8b-instant} \cite{llama2}. Groq proporciona una infraestructura de inferencia de ultra baja latencia (típicamente inferior a 500\,ms para el modelo 8B) mediante unidades de procesamiento LPU (\textit{Language Processing Unit}).

\textbf{Llama-3} pertenece a la familia de \textit{Large Language Models} (LLMs), que constituyen la aplicación más reciente del \textbf{aprendizaje profundo} (\textit{deep learning})~\cite{goodfellow2016}: redes neuronales con múltiples capas de transformación no lineal entrenadas sobre corpus de texto de escala masiva. Esta familia recoge la evolución histórica del campo desde los primeros sistemas expertos basados en reglas de los años 80 hasta las redes neuronales profundas actuales, pasando por los perceptrones multicapa (MLP) y la arquitectura \textit{Transformer}. El mecanismo de \textbf{atención} (\textit{attention}) que subyace a los Transformers fue propuesto por Vaswani et al.\ en 2017 en el artículo \textit{Attention Is All You Need}~\cite{vaswani2017}, y es el componente central que permite a los LLMs capturar dependencias de largo alcance en el texto. El uso de Llama-3 en FurniGest ilustra el paradigma del \textbf{\textit{transfer learning}}: un modelo preentrenado con billones de tokens es reutilizado en una tarea específica de razonamiento analítico mediante \textit{prompt engineering} sin necesidad de reentrenamiento \textit{fine-tuning}.

La integración sigue un patrón de \textit{prompt engineering} estructurado en dos capas:

\begin{enumerate}
  \item \textbf{Mensaje de sistema:} describe el rol del asistente como analista financiero experto de una tienda de muebles, indica el formato de respuesta esperado y define las restricciones (no inventar cifras, usar solo los datos inyectados).
  \item \textbf{Contexto de métricas:} antes de cada petición del usuario, el \textit{backend} calcula las métricas del periodo seleccionado (ventas, top productos, RFM, pares) y las serializa como JSON compacto, que se incorpora al historial de la conversación como mensaje de usuario previo.
\end{enumerate}

Esta arquitectura permite que el modelo realice razonamiento analítico sobre datos reales sin necesidad de un modelo de mayor tamaño: la tarea no es recuperación de conocimiento, sino razonamiento numérico sobre un JSON estructurado.

El Código~\ref{cod:groq} muestra la función de llamada a la API de Groq, que utiliza el cliente HTTP estándar de Python (\texttt{requests}) sobre el endpoint compatible con la API de OpenAI.

\begin{mypython}[float={!t},caption={Llamada al LLM mediante la API de Groq (\texttt{utils/groq\_llm.py}).},label={cod:groq}]
import requests
from backend.app.ia_settings import GROQ_API_KEY, GROQ_BASE_URL, GROQ_MODEL

def groq_chat(messages, temperature: float = 0.2,
              model: str | None = None) -> str:
    url = f"{GROQ_BASE_URL.rstrip('/')}/chat/completions"
    headers = {
        "Authorization": f"Bearer {GROQ_API_KEY}",
        "Content-Type": "application/json",
    }
    payload = {
        "model": model or GROQ_MODEL,
        "temperature": float(temperature),
        "messages": messages,
    }
    r = requests.post(url, headers=headers,
                      json=payload, timeout=60)
    r.raise_for_status()
    return r.json()["choices"][0]["message"]["content"].strip()
\end{mypython}

\subsection{Módulo de pagos con Stripe}

La integración de pagos sigue el flujo \textit{Stripe Checkout} \cite{stripe} en su modalidad con sesión creada en el servidor:

\begin{enumerate}
  \item El \textit{frontend} (\texttt{BancoPage}) envía un \texttt{POST /api/stripe/checkout} con el importe y la descripción del cobro.
  \item El \textit{backend} crea una sesión de Stripe con \texttt{payment\_method\_types=["card"]} y devuelve la URL de pago al cliente.
  \item El navegador redirige al usuario a la página hospedada de Stripe; al completarse el pago, Stripe devuelve al usuario a \texttt{/banco?success=true}.
  \item El \textit{frontend} llama a \texttt{POST /api/stripe/confirm} con el identificador de sesión.
  \item El \textit{backend} verifica el estado de la sesión (\texttt{payment\_status == "paid"}) mediante la API de Stripe y, si el pago está confirmado, crea un movimiento de tipo \texttt{INGRESO} y registra la sesión en \texttt{stripe\_checkouts} para garantizar idempotencia.
\end{enumerate}

Este diseño externaliza íntegramente la página de pago a la infraestructura de Stripe, eliminando cualquier obligación de cumplimiento PCI en el servidor de la aplicación. Las claves secretas de Stripe nunca se transmiten al navegador.


\subsection{Generación de documentos PDF}

FurniGest genera tres tipos de documentos PDF mediante la biblioteca \textbf{ReportLab~4.x} \cite{reportlab}:

\begin{itemize}
  \item \textbf{Albarán de cliente} (\texttt{utils/albaran\_pdf.py}): incluye datos del cliente, tabla de líneas con productos, cantidades y precios, total y estado. Se genera en memoria (\texttt{io.BytesIO}) y se adjunta al correo electrónico.
  \item \textbf{Factura de ruta} (\texttt{api/transportes.py}): lista los albaranes asignados al camión, con el total de la ruta y el coste de transporte (7\,\%).
  \item \textbf{Informe de analíticas} (\texttt{utils/tendencias\_pdf.py}): exporta las gráficas de ventas y los datos de segmentación RFM para su distribución o archivo.
\end{itemize}

Los endpoints de descarga de PDF devuelven \texttt{StreamingResponse} con \texttt{media\_type="application/pdf"}, lo que permite que el navegador abra directamente el documento sin necesidad de guardar fichero temporal en el servidor.


\subsection{Notificaciones por correo electrónico}

Al crear un albarán, el \textit{backend} encola en una \texttt{BackgroundTask} de FastAPI el envío de un correo con el PDF del albarán adjunto. El uso de \texttt{BackgroundTask} garantiza que la respuesta HTTP (201 Created) se devuelve al cliente de forma inmediata; el correo se envía de forma asíncrona después.

El módulo \texttt{utils/emailer.py} implementa una estrategia dual:

\begin{itemize}
  \item \textbf{Desarrollo local:} se usa \texttt{smtplib} sobre Gmail SMTP (puerto 587, STARTTLS) con una contraseña de aplicación de 16 caracteres generada en la cuenta de Google.
  \item \textbf{Producción (Railway):} Railway bloquea las conexiones SMTP salientes. Cuando se detecta la variable \texttt{RESEND\_API\_KEY} en el entorno, el módulo cambia automáticamente al proveedor \textbf{Resend} \cite{resend}, cuya API HTTP opera sobre el puerto 443, nunca bloqueado. El cuerpo HTML del mensaje y el PDF adjunto en \textit{base64} se envían a la API de Resend mediante una petición POST.
\end{itemize}

El cuerpo del correo se renderiza a partir de la plantilla Jinja2 \texttt{templates/albaran\_email.html}, que incluye los datos del cliente, el resumen del pedido y el importe total.


\subsection{Despliegue y CI/CD}
\label{sec:cicd}

La infraestructura de despliegue aplica los tres niveles del modelo de \textbf{entrega de software continua} \cite{fowlerCI}:

\begin{itemize}
  \item \textbf{Integración continua (CI):} cada \textit{push} a la rama \texttt{main} desencadena la ejecución automática de la batería de pruebas. Martin Fowler (2006) \cite{fowlerCI} define la integración continua como la práctica en la que ``cada miembro del equipo integra su trabajo al repositorio principal al menos una vez al día, siendo verificado por un proceso de \textit{build} automatizado que detecta errores lo antes posible''. En FurniGest, este proceso está orquestado por \textbf{GitHub Actions} \cite{githubactions}.
  \item \textbf{Entrega continua (\textit{Continuous Delivery}):} si todos los tests pasan, los artefactos compilados (imagen Docker del \textit{backend}, \_bundle\_ estático del \textit{frontend}) están listos para ser desplegados en cualquier momento con un único paso manual.
  \item \textbf{Despliegue continuo (\textit{Continuous Deployment}):} en FurniGest el despliegue es completamente automático, sin intervención humana. \textbf{Vercel} \cite{vercel} recibe el \textit{bundle} del \textit{frontend} (SPA estática generada por \texttt{vite build}) y lo sirve desde su CDN global. \textbf{Railway} \cite{railway} construye y arranca el contenedor Docker del \textit{backend} directamente desde GitHub, inyectando la variable \texttt{DATABASE\_URL} y gestionando la instancia PostgreSQL~16.
\end{itemize}

Para la ejecución local la aplicación completa puede levantarse con un único comando:

\begin{verbatim}
docker compose up --build
\end{verbatim}

Docker Compose \cite{docker} define tres servicios: \texttt{db} (PostgreSQL~16), \texttt{backend} (FastAPI + Uvicorn) y \texttt{frontend} (React compilado con Nginx). El servicio \texttt{backend} espera a que \texttt{db} pase su \textit{healthcheck} antes de iniciarse, evitando errores de conexión en el arranque.

\textbf{Docker} implementa el paradigma de \textbf{virtualización basada en contenedores}: a diferencia de las máquinas virtuales que emulan hardware completo mediante un \textit{hypervisor}, los contenedores comparten el núcleo del sistema operativo anfitrión aislando únicamente el espacio de procesos, sistema de ficheros y red~\cite{docker}. Esto reduce drásticamente el tiempo de arranque y el consumo de recursos, siendo especialmente adecuado para arquitecturas de servicios distribuidos como la de FurniGest.

En producción, toda la comunicación entre el cliente y el servidor se protege con \textbf{TLS} (\textit{Transport Layer Security})~\cite{kurose} mediante el certificado gestionado automáticamente por Railway. TLS cifra el tráfico HTTP mediante \textbf{AES} (cifrado simétrico de sesión) y garantiza la autenticidad del servidor a través de certificados \textbf{X.509}, cumpliendo los principios de confidencialidad e integridad de la tríada CIA establecida en el RNF-06.

\subsection{Desarrollo colaborativo con Git}
\label{sec:git_colaborativo}

El modelo de ramas de FurniGest sigue el enfoque \textbf{Trunk-Based Development} (TBD)~\cite{tbd}: la rama \texttt{main} es la única rama de larga duración; cada funcionalidad se desarrolla en una \textit{feature branch} de vida corta que se integra mediante un \textit{Pull Request} (PR). AIS T3.2 presenta dos alternativas principales:

\begin{table}[H]
  \centering
  \caption{Comparativa de modelos de ramas: Gitflow vs.\ Trunk-Based Development.}
  \label{tab:branching}
  \renewcommand{\arraystretch}{1.4}
  \begin{footnotesize}
  \begin{tabular}{|l|p{4.5cm}|p{4.5cm}|}
    \hline
    & \textbf{Gitflow} & \textbf{Trunk-Based (TBD)} \\
    \hline
    Ramas permanentes & \texttt{main} + \texttt{develop} + \texttt{release/*} + \texttt{hotfix/*} & Solo \texttt{main} \\
    Vida de feature branch & Días o semanas & $<$ 1--2 días \\
    Releases & Periódicas, coordinadas & Continuas, cada PR \\
    Coste de integración & Alto (merges frecuentes entre ramas) & Bajo (integración diaria) \\
    Adecuado para & Proyectos con ciclos de versión fijos & Entrega continua (CD) \\
    \hline
  \end{tabular}
  \end{footnotesize}
\end{table}

Se eligió TBD porque FurniGest aplica \textbf{despliegue continuo}: cada PR mergeado a \texttt{main} activa automáticamente el workflow \texttt{deploy.yml}, publicando la nueva versión sin intervención humana. Gitflow añadiría la rama \texttt{develop} y ramas de release sin beneficio en un proyecto unipersonal.

Todos los mensajes de commit siguen la especificación \textbf{Conventional Commits}~\cite{conventionalcommits}, que estandariza el formato del asunto:

\begin{verbatim}
<tipo>[ámbito opcional]: <descripción imperativa en minúsculas>

[cuerpo opcional — explica QUÉ y POR QUÉ, no cómo]
\end{verbatim}

Los tipos usados en FurniGest son: \texttt{feat} (nueva funcionalidad), \texttt{fix} (corrección de bug), \texttt{refactor}, \texttt{test}, \texttt{docs} y \texttt{chore} (mantenimiento). Esta convención facilita la generación automática de \textit{changelogs} y hace que el historial sea autoexplicativo~\cite{beams}.

Cada PR rellena una plantilla estructurada (\texttt{.github/PULL\_REQUEST\_TEMPLATE.md}) que incluye: descripción del cambio, tipo de commit, lista de ficheros modificados, confirmación de que los tests locales pasan y confirmación de que el pipeline de CI es verde. Esto garantiza que ningún commit roto llegue a producción, alineándose con la definición de \textbf{Integración Continua} de Fowler~\cite{fowlerCI}.


\section{Pruebas}

La estrategia de pruebas de FurniGest sigue la \textbf{pirámide de pruebas} \cite{cohn}: una base amplia de \textbf{pruebas unitarias}, una capa intermedia de \textbf{pruebas de integración} que verifican la interacción entre componentes, y un número reducido de \textbf{pruebas de sistema} que validan la aplicación completa desde el punto de vista del usuario. Las pruebas automatizadas actúan como \textbf{red de seguridad frente a regresiones}: cualquier cambio en el código que rompa comportamiento existente es detectado en segundos durante el pipeline de CI antes de que llegue a producción.

\subsection{Pruebas del backend}

Las pruebas del \textit{backend} son \textbf{pruebas de integración HTTP} implementadas con \textbf{pytest} \cite{pytest} y \texttt{TestClient} de FastAPI, que ejerce la aplicación completa (enrutador $\to$ lógica de negocio $\to$ base de datos) sin levantar un servidor real. El \textbf{SUT} (\textit{System Under Test}) en cada caso de prueba es el endpoint completo. La configuración utiliza una base de datos SQLite en memoria para garantizar el \textbf{aislamiento} entre pruebas y la repetibilidad de los resultados sin necesidad de una instancia PostgreSQL real en el entorno de CI. \subsubsection{Dobles de test en servicios externos}

Varios endpoints de FurniGest llaman a servicios externos: la API de \textbf{Groq} (\textit{analytics}), la API de \textbf{Stripe} (\textit{pagos}) y el servicio de \textbf{email} (\textit{albaranes}). Invocar esas APIs reales en los tests violaría el principio \textbf{FIRST} de las pruebas unitarias~\cite{cohn}: los tests dejarían de ser \textit{Isolated} (dependerían de keys externas) y de ser \textit{Fast} (latencia de red).

Para resolver esto, FurniGest utiliza \textbf{dobles de test} (\textit{test doubles}), según la taxonomía de Gerard Meszaros (\textit{xUnit Test Patterns}), implementados con la librería \textbf{pytest-mock}:

\begin{table}[H]
  \centering
  \caption{Tipos de doble de test usados en FurniGest.}
  \label{tab:test_doubles}
  \renewcommand{\arraystretch}{1.4}
  \begin{footnotesize}
  \begin{tabular}{|l|p{4.5cm}|p{5cm}|}
    \hline
    \textbf{Tipo} & \textbf{Definición} & \textbf{Uso en FurniGest} \\
    \hline
    \textit{Stub}  & Devuelve valores fijos; no verifica llamadas & \texttt{groq\_chat} devuelve un informe predefinido en los tests de analíticas \\
    \textit{Mock}  & Verifica que fue llamado con los parámetros correctos & \texttt{stripe.checkout.Session.create} verificado exactamente 1 vez \\
    \textit{Spy}   & Como mock pero también ejecuta la implementación real & \texttt{mock.assert\_called\_once()} sobre la sesión de Stripe \\
    \textit{Fake}  & Implementación simplificada & SQLite en memoria como sustituto de PostgreSQL \\
    \textit{Dummy} & Objeto vacío, nunca se usa realmente & Respuesta vacía de email (\texttt{return\_value=None}) \\
    \hline
  \end{tabular}
  \end{footnotesize}
\end{table}

Los tests siguen la estructura \textbf{Given / When / Then} (AIS T1.2): el bloque \textit{Given} configura el doble con \texttt{mocker.patch()}, el bloque \textit{When} invoca el endpoint, y el bloque \textit{Then} verifica tanto la respuesta HTTP como las llamadas al doble (\texttt{mock.assert\_called\_once()}).

\subsection{Calidad del código y análisis estático}

El \textbf{análisis estático de código (SCA, \textit{Static Code Analysis})} \cite{fowlerRefactoring} consiste en examinar el código fuente automáticamente sin ejecutarlo, con el objetivo de detectar errores potenciales, violaciones de estilo, código duplicado y posibles vulnerabilidades de seguridad. El repositorio aplica dos herramientas de SCA:

\begin{itemize}
  \item \textbf{Ruff} \cite{ruff}: \textit{linter} y formateador de código Python ultrarrápido que agrupa las reglas de PEP~8, Pyflakes y otras más de 500 reglas de estilo. Se verifica automáticamente en el workflow \texttt{lint.yml} de GitHub Actions. El proyecto mantiene \textbf{cero avisos} de Ruff en todo momento, lo que garantiza una \textbf{calidad interna} mínima del código fuente \cite{fowlerRefactoring}: legibilidad, eliminación de importaciones no utilizadas y ausencia de patrones anti-idiomáticos.
  \item \textbf{ESLint}: análisis estático de JavaScript/JSX con las reglas \texttt{eslint:recommended} y \texttt{react-hooks/recommended}, que detectan dependencias incorrectas en efectos de React y uso inadecuado de los Hooks. Integrado en el workflow de CI.
\end{itemize}

La ausencia de \textit{code smells} ---indicadores de degradación de la calidad interna identificados por Martin Fowler \cite{fowlerRefactoring}, como funciones demasiado largas, nombres poco expresivos o código duplicado--- se mantiene mediante refactorizaciones periódicas durante el desarrollo siguiendo la \textit{Regla del Boy Scout}: dejar el código siempre en mejor estado del que se encontró.

Esta práctica de análisis estático continuo es coherente con la filosofía de \textbf{desarrollo seguro} que promueve OWASP~\cite{owasp2021}: «concebir, desarrollar y mantener aplicaciones confiables». Ruff y ESLint actúan como herramientas SCA \textit{shift-left}, detectando defectos potenciales en el momento del \textit{commit} antes de que lleguen a producción, en la misma línea que herramientas del ecosistema OWASP como PMD, FlawFinder o LAPSE.

\chapter{Conclusiones y trabajos futuros}

\section{Logros alcanzados}

FurniGest cumple íntegramente los doce objetivos específicos enunciados en el Capítulo~2. La aplicación está desplegada en producción, es accesible desde cualquier navegador web y ha sido validada con una batería de 157~pruebas automatizadas que se ejecutan en cada \textit{commit}.

Desde el punto de vista funcional, el sistema transforma los flujos de trabajo de la tienda en procesos digitales completos: la creación de un pedido activa simultáneamente el registro contable, la generación del PDF y el envío del correo al cliente; la entrega de los artículos liquida automáticamente el saldo pendiente; y la asignación de un camión crea el albarán de transporte y registra su coste sin intervención manual adicional.

Desde el punto de vista técnico, el proyecto ha demostrado la viabilidad de construir un ERP a medida de calidad profesional con tecnologías de código abierto y servicios en la nube con coste de operación nulo o muy reducido:

\begin{itemize}
  \item \textbf{FastAPI + SQLAlchemy} han permitido construir una API REST sólida y bien documentada con muy pocas líneas de código gracias a la generación automática de esquemas OpenAPI y la validación de datos mediante Pydantic.
  \item \textbf{React~19 + TailwindCSS~4} han posibilitado una interfaz moderna, responsiva y mantenible, con un proceso de compilación en segundos gracias a Vite.
  \item La integración de \textbf{Groq/Llama-3} como asistente de IA ha resultado especialmente efectiva al inyectar métricas reales en el contexto del modelo: el sistema no requiere un LLM de gran tamaño porque el razonamiento se produce sobre datos estructurados, no sobre conocimiento general.
  \item La arquitectura de \textbf{CI/CD} con GitHub Actions garantiza que ningún cambio defectuoso llegue a producción.
\end{itemize}

\section{Conclusiones técnicas}

El desarrollo de FurniGest ha confirmado que el patrón de inyección de dependencias de FastAPI (\texttt{Depends(get\_db)}) es extraordinariamente eficaz para gestionar recursos por petición sin compartir estado entre \textit{workers}. Las \texttt{BackgroundTask} de FastAPI resultan suficientes para tareas de vida corta como el envío de correos, evitando la complejidad operacional de Celery. El modelo de eventos del ciclo de vida (\textit{lifespan}) es la forma idiomática de gestionar la inicialización de la aplicación en las versiones recientes de FastAPI.

En el \textit{frontend}, el patrón \texttt{useState} + \texttt{useEffect} de React ha sido suficiente para el estado local de cada componente. No ha sido necesario ningún gestor de estado global (Redux, Zustand) porque el estado compartido entre páginas es mínimo y se gestiona mediante \texttt{localStorage} para los casos puntuales de datos de navegación cruzada.

\section{Trabajos futuros}

Las siguientes líneas de trabajo ampliarían significativamente las capacidades de FurniGest:

\begin{enumerate}
  \item \textbf{Autenticación y control de acceso:} FurniGest opera actualmente sin capa de autenticación. La incorporación de \textit{JSON Web Tokens} (JWT) con FastAPI Security permitiría diferenciar roles (administrador, vendedor, repartidor) y auditar las acciones de cada usuario.

  \item \textbf{Aplicación móvil nativa:} la API REST existente podría consumirse desde una aplicación \textit{React Native} o \textit{Flutter} que mejore la experiencia de los repartidores al gestionar las entregas desde un teléfono en campo.

  \item \textbf{Gestión de stock e inventario:} actualmente el catálogo de productos almacena precios y descripciones pero no un nivel de stock real. La extensión del modelo con un campo de inventario, alertas de stock mínimo y un flujo de recepción de mercancía completaría el ciclo de producto.

  \item \textbf{Módulo de presupuestos:} incorporar una fase previa al albarán (presupuesto aceptado/rechazado por el cliente) mejoraría la trazabilidad comercial y la previsión de ventas.

  \item \textbf{Analíticas predictivas:} ampliar el motor de analíticas con modelos de previsión de demanda (ARIMA, Prophet) o predicción de \textit{churn} de clientes aprovecharía los datos históricos ya acumulados.

  \item \textbf{Notificaciones \textit{push}:} integrar Web Push API o un canal de Slack/Telegram para notificaciones en tiempo real (nuevo pedido, entrega confirmada, pago recibido) reduciría la dependencia del correo electrónico.

  \item \textbf{Internacionalización (i18n):} la aplicación podría adaptarse fácilmente a otros sectores e idiomas parametrizando el vocabulario de dominio (albarán, fianza, camión) mediante un sistema de traducciones como \textit{i18next}.
\end{enumerate}

    nombre           VARCHAR NOT NULL,
    apellidos        VARCHAR NOT NULL,
    dni              VARCHAR UNIQUE,
    email            VARCHAR,
    telefono1        VARCHAR,
    telefono2        VARCHAR,
    calle            VARCHAR,
    numero_vivienda  VARCHAR,
    piso_portal      VARCHAR,
    ciudad           VARCHAR,
    codigo_postal    VARCHAR
);

-- Tabla de proveedores
CREATE TABLE proveedores (
    id      SERIAL PRIMARY KEY,
    nombre  VARCHAR NOT NULL,
    email   VARCHAR,
    telefono VARCHAR
);

-- Tabla de productos
CREATE TABLE productos (
    id            SERIAL PRIMARY KEY,
    nombre        VARCHAR NOT NULL,
    descripcion   VARCHAR,
    precio        FLOAT DEFAULT 0.0,
    proveedor_id  INTEGER REFERENCES proveedores(id)
);

-- Tabla de albaranes
CREATE TABLE albaranes (
    id           SERIAL PRIMARY KEY,
    fecha        DATE NOT NULL,
    descripcion  VARCHAR,
    total        FLOAT DEFAULT 0.0,
    estado       VARCHAR DEFAULT 'FIANZA' NOT NULL,
    cliente_id   INTEGER REFERENCES clientes(id)
);

-- Tabla de líneas de albarán
CREATE TABLE lineas_albaran (
    id               SERIAL PRIMARY KEY,
    albaran_id       INTEGER REFERENCES albaranes(id) ON DELETE CASCADE,
    producto_id      INTEGER REFERENCES productos(id),
    cantidad         INTEGER NOT NULL,
    precio_unitario  FLOAT NOT NULL
);

-- Tabla de movimientos financieros
CREATE TABLE movimientos (
    id        SERIAL PRIMARY KEY,
    fecha     DATE NOT NULL,
    concepto  VARCHAR NOT NULL,
    cantidad  FLOAT NOT NULL,
    tipo      VARCHAR NOT NULL  -- 'INGRESO' | 'GASTO'
);

-- Tabla de rutas de transporte
CREATE TABLE albaran_rutas (
    id          SERIAL PRIMARY KEY,
    albaran_id  INTEGER UNIQUE REFERENCES albaranes(id),
    camion_id   INTEGER NOT NULL
);

-- Tabla de registros Stripe
CREATE TABLE stripe_checkouts (
    id          SERIAL PRIMARY KEY,
    session_id  VARCHAR UNIQUE NOT NULL,
    amount      FLOAT NOT NULL,
    description VARCHAR,
    status      VARCHAR NOT NULL,
    created_at  DATE
);
